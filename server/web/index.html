under the hood

How encryption works

step by step
  1. Your file bytes arrive at the server over HTTPS
  2. Server generates a fresh AES-256-GCM key + random 96-bit nonce
  3. File is encrypted → ciphertext. Plaintext is immediately discarded from memory
  4. Key is stored in the vault with your unlock timestamp. Only the key is stored — not your file
  5. Response: a .tlp file containing key_id + ciphertext. This is what you keep
  6. Server returns the key only after the unlock date — no exceptions, no admin override
💡

The .tlp file holds only ciphertext + a key reference ID. Even if someone intercepts your .tlp, they cannot decrypt it without the server releasing the key.

.tlp format

What's inside a .tlp file

json structure
{
  "v": 1,
  "id": "r4nd0m_key_id", // server-side key ref
  "unlock_at": 1798761600, // unix timestamp
  "unlock_iso": "2027-01-01T00:00:00+00:00",
  "original": "file.txt",
  "nonce": "base64...", // AES-GCM nonce
  "ct": "base64...", // ciphertext
  "_sig": "integrity_hash" // tamper check
}
🔑

Send your .tlp file here. If the unlock date has passed the server releases the key and returns your original file. If not — you get a countdown and HTTP 423 Locked.

under the hood

How decryption works

step by step
  1. Your .tlp file is sent to the server — contains a key_id and the ciphertext
  2. Server verifies the _sig integrity field — rejects tampered files
  3. Server checks current time against unlock_at timestamp
  4. If locked → returns HTTP 423 with countdown JSON. Request ends here
  5. If unlocked → server looks up the AES-256 key by key_id from the vault
  6. Ciphertext decrypted with AES-256-GCM → original file bytes returned
⚠️

Key persistence: Keys are stored indefinitely. If you self-host, ensure your vault.db is not deleted before your files need to be unlocked.
